This is an Apache 2.0 Nx monorepo that produces the `@adaptiveworx/iac-*`
family of npm packages. It's designed for agentic development: rules
below apply equally to Claude Code, Codex, Cursor, or any agent.

Claude-specific guidance lives in [`CLAUDE.md`](./CLAUDE.md) — read it
in addition if you're Claude.

## The rule that overrides everything

**`iac-core` (the package) is cloud-agnostic only.** Full statement
of the boundary lives in
[`docs/architecture.md#boundary-rules`](./docs/architecture.md#boundary-rules).
Default test before adding code: *would an Azure-only consumer use
this without contortions?* If no, it belongs in a cloud-specific
package (`iac-aws`, `iac-azure`).

There is **no `iac-shared` package.** Don't propose one.

## Releases — never hand-edit versions

Versioning is conventional-commit-driven via Nx Release. **Never edit
a `version` field in any `package.json` by hand.** A pre-commit hook
will reject the change anyway. `feat:` → minor, `fix:` → patch,
`feat!:` / `BREAKING CHANGE:` → major. `chore:`, `ci:`, `docs:`,
`refactor:`, `test:`, `style:`, `build:` never bump.

Full release flow: [`CONTRIBUTING.md#releases`](./CONTRIBUTING.md#releases).

## Conventional commits — scope rules

Every commit must follow Conventional Commits. The commit-msg hook
runs `commitlint` and rejects malformed messages.

**Scope allowlist:** `iac-core`, `iac-schemas`, `iac-policies`,
`iac-aws`, `iac-azure`, or `repo`. Exact rules:

- `feat:` and `fix:` **always** carry a single package scope (e.g.
  `feat(iac-aws):`). They will bump that package's version — the scope
  tells Nx Release which one.
- Cross-cutting work (tooling, monorepo plumbing, repo docs) uses
  scope `repo` AND a non-bumping type (`chore(repo):`, `ci(repo):`,
  `docs(repo):`, etc.). These shouldn't bump anything.
- A single commit changes a single package surface. If you have to
  touch two packages for one logical change, split into two commits —
  one per scope.
- Never use `feat(repo):` or `fix(repo):`. Version-bumping types
  attached to a non-package scope trigger file-touch fallback across
  every package.

## Use `nx affected`, not full-repo runs

For local validation, prefer the affected graph:

```bash
pnpm nx affected -t lint typecheck test build --base=origin/main --head=HEAD
```

The pre-push hook runs this automatically. CI uses `nx affected` on
PRs and `nx run-many` on main as the authoritative backstop. Reach
for `nx run-many` locally only when you're verifying the whole repo
intentionally (e.g. before a release).

## What not to touch

- **`packages/*/package.json` `version` fields.** Owned by Nx Release.
- **`packages/*/CHANGELOG.md`.** Generated by Nx Release.
- **`.github/workflows/release.yml`.** Holds publish authority — only
  change with a clear reason and surface the why in the PR description.
- **`nx.json` `release` block.** Versioning + release config.
- **`pnpm-lock.yaml`.** Touch only via `pnpm install`; never hand-edit.
- **`iac-worx/libs/iac/*` source from this repo's session.** Mid-migration;
  fix in `iac-worx` first, then carry into the migrated package here.

## Pre-flight before declaring a task done

Hooks catch the cheap stuff. Before saying "done" on substantive work:

1. `pnpm nx affected -t lint typecheck test build` (or `:affected` script aliases) — must be clean.
2. New public exports → reflected in the package's README.
3. Cross-cutting docs touched → links resolve, references aren't stale.
4. `git status` clean. No stray `.env`, `dist/`, or generated artifacts.

If you skipped a pre-push hook with `LEFTHOOK=0` for some reason, run
the affected check manually before declaring done.

## Public-repo discipline

This repo is published under `@adaptiveworx` on public npm.

- No client-confidential identifiers (account IDs not already public,
  internal IPs, secrets, internal infra hostnames).
- Existing hardcoded values (e.g. AdaptiveWorX-shaped defaults) are
  tracked as debt issues — see [`docs/migration-plan.md`](./docs/migration-plan.md).
  Don't add new ones.
- Prosilio is a named consumer in existing docs — that's precedent and
  fine. New client names need a separate decision.

## Don't add packages or top-level dirs without updating docs

New `packages/*` entry → update
[`docs/architecture.md`](./docs/architecture.md) package table,
[`README.md`](./README.md) package table, and add to the workflow
allowlist in `.github/workflows/release.yml` and the commitlint scope
allowlist in `commitlint.config.cjs`.

New top-level dir → mention in `docs/architecture.md`.

If you're not sure whether something needs documenting, it does.
